> **Exception Chaining is the process of wrapping one exception inside another exception while preserving the original cause.**

In simple words:

> **One exception causes another exception.**

Instead of losing the original exception, we attach it as the **cause** of the new exception.

---

# Why Do We Need Exception Chaining?

Suppose your application has multiple layers:

```
Presentation Layer
        |
Service Layer
        |
DAO Layer
        |
Database
```

Imagine the database throws:

```
SQLException
```

The Service Layer doesn't want to expose database-specific exceptions to the UI, so it throws:

```
ServiceException
```

If we simply write:

```
throw new ServiceException("Unable to process request");
```

the original `SQLException` is lost.

Instead, we chain it:

```
throw new ServiceException("Unable to process request", e);
```

Now both exceptions are available.

---

# Real-Life Analogy

Imagine a car doesn't start.

```
Car Won't Start
       |
Battery Dead
       |
Alternator Failed
```

The visible problem is:

> Car won't start.

The **root cause** is:

> Alternator failed.

Exception chaining preserves the root cause.

# Without Exception Chaining

```
import java.sql.SQLException;

public class Demo {

    static void dao() throws SQLException {

        throw new SQLException("Database Connection Failed");

    }

    static void service() throws Exception {

        try {

            dao();

        }

        catch(SQLException e){

            throw new Exception("Service Failed");

        }

    }

    public static void main(String[] args){

        try{

            service();

        }

        catch(Exception e){

            e.printStackTrace();

        }

    }

}
```

Output

```
java.lang.Exception: Service Failed
```

### Problem

The original `SQLException` is completely lost.

---

# With Exception Chaining

```
import java.sql.SQLException;

public class Demo {

    static void dao() throws SQLException {

        throw new SQLException("Database Connection Failed");

    }

    static void service() throws Exception {

        try {

            dao();

        }

        catch(SQLException e){

            throw new Exception("Service Failed", e);

        }

    }

    public static void main(String[] args){

        try{

            service();

        }

        catch(Exception e){

            e.printStackTrace();

        }

    }

}
```

Output

```
java.lang.Exception: Service Failed

Caused by:
java.sql.SQLException: Database Connection Failed
```

Now we know both:

- What happened?
- Why it happened?

# Constructor Used

Most exception classes provide constructors like:

```
Exception(String message, Throwable cause)
```

Example

```
throw new Exception("Service Failed", e);
```

Where

```
Message : Service Failed

Cause : SQLException
```
