# What is a finally Block?

A **finally** block is used to write code that **should execute regardless of whether an exception occurs or not**.

It is primarily used for **resource cleanup**, such as:

- Closing files
- Closing database connections
- Closing sockets
- Closing streams
- Releasing locks

---

# Syntax

```
try {
    // Risky code
}
catch(Exception e) {
    // Exception handling
}
finally {
    // Cleanup code
}
```

You can also have:

```
try {
    // Risky code
}
finally {
    // Cleanup code
}
```

---

# Execution Flow

```
              Start
                 |
                 V
              try Block
                 |
          Exception Occurred?
            /             \
          Yes             No
           |               |
     Matching catch?       |
       /         \          |
     Yes         No         |
      |           |         |
 Execute catch    |         |
      |           |         |
      +-----------+---------+
                  |
             finally Block
                  |
      Unhandled exception?
           /          \
         Yes          No
          |            |
     JVM Terminates  Continue
```

# Case 1: No Exception

```
public class Demo {

    public static void main(String[] args) {

        try {
            System.out.println("Inside Try");
        }
        finally {
            System.out.println("Inside Finally");
        }

    }

}
```

### Output

```
Inside Try
Inside Finally
```

---

# Case 2: Exception Handled

```
public class Demo {

    public static void main(String[] args) {

        try {

            int x = 10 / 0;

        }
        catch (ArithmeticException e) {

            System.out.println("Handled");

        }
        finally {

            System.out.println("Finally");

        }

    }

}
```

### Output

```
Handled
Finally
```

---

# Case 3: Exception Not Handled

```
public class Demo {

    public static void main(String[] args) {

        try {

            int x = 10 / 0;

        }
        finally {

            System.out.println("Finally");

        }

    }

}
```

### Output

```
Finally

Exception in thread "main"
java.lang.ArithmeticException: / by zero
```

### Execution Table

|Step|Statement|Result|
|---|---|---|
|1|Enter `try`|✔|
|2|`10 / 0`|Exception occurs|
|3|Execute `finally`|✔|
|4|No `catch` found|Exception propagates|
|5|Program terminates|✔|

# Case 4: Return in try

```
public class Demo {

    static int test() {

        try {

            return 10;

        }
        finally {

            System.out.println("Finally");

        }

    }

    public static void main(String[] args) {

        System.out.println(test());

    }

}
```

### Output

```
Finally
10
```

### Explanation

Before the method actually returns, the JVM executes the `finally` block.

---

# Case 5: Return in catch

```
public class Demo {

    static int test() {

        try {

            int x = 10 / 0;
            return 1;

        }
        catch (ArithmeticException e) {

            return 2;

        }
        finally {

            System.out.println("Finally");

        }

    }

    public static void main(String[] args) {

        System.out.println(test());

    }

}
```

### Output

```
Finally
2
```

---

# Case 6: Return in finally (Very Important)

```
public class Demo {

    static int test() {

        try {

            return 10;

        }
        finally {

            return 20;

        }

    }

    public static void main(String[] args) {

        System.out.println(test());

    }

}
```

### Output

```
20
```

### Why?

Execution order:

1. `try` computes `return 10`.
2. Before returning, `finally` executes.
3. `finally` also returns a value.
4. The return from `finally` **overrides** the return from `try`.

---

# Dry Run

```
static int test() {

    try {
        return 100;
    }
    finally {
        return 200;
    }

}
```

### Dry Run Table

|Step|Action|
|---|---|
|Enter try|✔|
|Compute `return 100`|✔|
|Execute finally|✔|
|`return 200`|✔|
|Method returns|200|

# Case 7: Exception in try, return in finally

```
public class Demo {

    static int test() {

        try {

            int x = 10 / 0;

        }
        finally {

            return 50;

        }

    }

    public static void main(String[] args) {

        System.out.println(test());

    }

}
```

### Output

```
50
```

### Explanation

The `ArithmeticException` is **suppressed** because the `return` in `finally` takes precedence.

> **Interview Tip:** Returning from `finally` is strongly discouraged because it can hide exceptions and make debugging difficult.

---

# Case 8: Exception in finally

```
public class Demo {

    public static void main(String[] args) {

        try {

            System.out.println("Try");

        }
        finally {

            throw new RuntimeException("Error in Finally");

        }

    }

}
```

### Output

```
Try

Exception in thread "main"
RuntimeException: Error in Finally
```

If both the `try` block and the `finally` block throw exceptions, the exception from the `finally` block is the one that propagates (unless special handling such as suppressed exceptions is used in try-with-resources).

---

# Case 9: System.exit()

```
public class Demo {

    public static void main(String[] args) {

        try {

            System.out.println("Try");

            System.exit(0);

        }
        finally {

            System.out.println("Finally");

        }

    }

}
```

### Output

```
Try
```

### Explanation

`System.exit(0)` terminates the JVM immediately, so the `finally` block is **not executed**.

---

# When Does finally NOT Execute?

Although people often say "finally always executes," there are important exceptions.

|Situation|finally Executes?|
|---|---|
|No exception|✅ Yes|
|Exception handled|✅ Yes|
|Exception not handled|✅ Yes|
|`return` in try|✅ Yes|
|`return` in catch|✅ Yes|
|`throw` in try|✅ Yes|
|`throw` in catch|✅ Yes|
|`System.exit()`|❌ No|
|JVM crash|❌ No|
|Process killed by OS|❌ No|

# Best Practices

### ✔ Use finally for cleanup

```
Scanner scanner = new Scanner(System.in);

try {

    // use scanner

}
finally {

    scanner.close();

}
```

---

### ✔ Avoid business logic in finally

Keep the `finally` block focused on cleanup tasks.

---

### ✔ Do not return from finally

Bad:

```
finally {
    return 10;
}
```

Good:

```
finally {
    connection.close();
}
```

Returning from `finally` can override earlier return values and suppress exceptions.

---

# Common Interview Questions

### 1. Does finally always execute?

**Answer:** Almost always. It executes even if an exception occurs or a `return` statement is encountered, but it does **not** execute if the JVM terminates abruptly (for example, `System.exit(0)`), crashes, or is forcibly stopped.

---

### 2. Can we have try without catch?

**Yes.**

```
try {

}
finally {

}
```

---

### 3. Can we have finally without try?

**No.** A `finally` block must always be associated with a `try` block.

---

### 4. Can finally change the return value?

**Yes.**

A `return` statement in `finally` overrides returns from `try` and `catch`.

---

### 5. Can finally throw an exception?

**Yes.**

If it throws an exception, that exception propagates and may replace an exception that was already being thrown from the `try` block.

---

### 6. What is finally mainly used for?

To release resources such as files, database connections, sockets, streams, and locks.

---

### 7. Is finally required?

**No.**

You can write:

```
try {

}
catch(Exception e){

}
```

or

```
try {

}
finally{

}
```

# One-Minute Interview Revision

|Scenario|finally Executes?|
|---|---|
|Normal execution|✅ Yes|
|Exception handled|✅ Yes|
|Exception unhandled|✅ Yes|
|`return` in try|✅ Yes|
|`return` in catch|✅ Yes|
|`throw` in try|✅ Yes|
|`System.exit(0)`|❌ No|
|JVM crash|❌ No|

### Golden Rule

> The `finally` block is intended for **cleanup code** and executes **before a method actually returns or before an exception leaves the method**. Avoid returning from `finally`, because it can override return values and suppress exceptions, making programs harder to understand and debug.
